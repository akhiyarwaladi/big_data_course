Konsep Inti

Batch vs Stream Processing

Batch (P9-12)

  • Dikumpulkan, proses SEKALIGUS
  • Latensi: menit - jam
  • Contoh: laporan harian, ETL, dbt
  • Analogi: mesin cuci (kumpulkan, cuci sekaligus)
  • Tools: DuckDB, Spark, dbt

Stream (P13)

  • Proses SAAT DATA TIBA (real-time)
  • Latensi: milidetik - detik
  • Contoh: notifikasi, fraud, surge pricing
  • Analogi: conveyor belt (proses sambil jalan)
  • Tools: Kafka, Flink, Spark Streaming
!

Arsitektur

Apache Kafka: Arsitektur Dasar

1
Producer
Aplikasi yang MENGIRIM event/pesan ke Kafka (sensor, API, log server)
2
Topic
Channel/kategori event. Seperti "inbox" untuk jenis pesan tertentu
3
Consumer
Service yang MEMBACA dan memproses event dari topic

Decoupled! Producer dan Consumer tidak perlu tahu satu sama lain. Banyak consumer bisa baca 1 topic.

Konsep Kafka

Konsep-Konsep Penting Kafka

Producer

Aplikasi pengirim: sensor IoT, API endpoint, log server, mobile app

Topic

Channel event berdasarkan kategori: "rides", "payments", "clicks"

Consumer

Service pemroses: fraud detector, pricing engine, analytics pipeline

Offset

Bookmark posisi baca. Bisa replay dari awal atau lanjut dari terakhir

Consumer Group

Beberapa consumer berbagi beban. 1 partisi = 1 consumer dalam group

Partition

Topic dipecah jadi partisi untuk parallelism. Urutan terjaga per partisi

W

Windowing

3 Jenis Windowing

1
Tumbling Window
Fixed size, TIDAK overlap. [0-5m] [5-10m] [10-15m]. Setiap event masuk tepat 1 window. Contoh: hitung order per 5 menit.
2
Sliding Window
Fixed size, OVERLAP. Window 5m, slide 1m. Setiap event bisa masuk beberapa window. Contoh: moving average harga.
3
Session Window
Ukuran dinamis, berdasarkan GAP. Jika idle > threshold, window baru dimulai. Contoh: sesi browsing user.

Pattern

Event Sourcing

Traditional (State)

  • Simpan SALDO AKHIR: Rp 1.000.000
  • Update langsung ke database
  • Histori hilang setelah update
  • Tidak bisa "replay" kejadian
  • Sulit audit trail

Event Sourcing

  • Simpan SEMUA transaksi sebagai event
  • +5M, -2M, +3M, -1M, ... = saldo
  • Replay: hitung ulang dari event pertama
  • Audit trail lengkap otomatis
  • Time travel: lihat saldo kapan saja

Kafka = event log permanen. Semua event disimpan, bisa di-replay kapan saja.

Konsep

Delivery Guarantees

1
At-Most-Once
Kirim sekali, tidak ada retry. Bisa HILANG. Paling cepat, untuk data non-kritis (log, metrics).
2
At-Least-Once
Retry sampai berhasil. Bisa DUPLIKAT. Consumer harus idempotent. Default Kafka.
3
Exactly-Once
Tepat sekali, tidak hilang, tidak duplikat. Paling lambat & kompleks. Untuk payment/financial.

Trade-off: Semakin kuat guarantee, semakin lambat throughput. Pilih sesuai kebutuhan bisnis!

Simulasi Kafka dengan Python

from collections import defaultdict import threading class MessageBroker: """Simulasi Kafka: topics, offsets, batch consume""" def __init__(self): self.topics = defaultdict(list) self.offsets = defaultdict(int) self._lock = threading.Lock() def produce(self, topic, event): with self._lock: event['_offset'] = len(self.topics[topic]) self.topics[topic].append(event) def consume(self, topic, group='default', batch=100): with self._lock: key = f"{topic}:{group}" start = self.offsets[key] events = self.topics[topic][start:start+batch] self.offsets[key] = start + len(events) return events broker = MessageBroker()

Thread-safe + batch consume + offset tracking. Konsep 100% sama dengan Kafka asli.

Events bisa masuk beberapa window.""" if not events: return {} results = {} min_ts = min(e["timestamp"] for e in events) max_ts = max(e["timestamp"] for e in events) t = min_ts while t <= max_ts: w_events = [e for e in events if t <= e["timestamp"] < t + window_sec] if w_events: avg = sum(e["fare"] for e in w_events) / len(w_events) results[t] = {"count": len(w_events), "avg_fare": avg} t += slide_sec return results result = sliding_window(events) for t, v in list(result.items())[:5]: print(f"Window {t:.0f}: {v['count']} events, avg=Rp{v['avg_fare']:,.0f}")

Surge Pricing Detector

def detect_surge(events, window_sec=60, threshold=10): """Deteksi surge: jika demand > threshold dalam window per zona""" zone_windows = defaultdict(lambda: defaultdict(list)) for e in events: w = int(e["timestamp"] // window_sec) * window_sec zone_windows[e["zone"]][w].append(e) surges = [] for zone, windows in zone_windows.items(): for w, w_events in windows.items(): demand = len(w_events) if demand >= threshold: multiplier = 1.0 + (demand - threshold) * 0.1 surges.append({"zone": zone, "demand": demand, "surge": round(multiplier, 1)}) return surges surges = detect_surge(events) for s in surges: print(f"SURGE {s['zone']}: {s['demand']} orders, {s['surge']}x")

Event Sourcing: Shopping Cart

class CartEventStore: def __init__(self): self.events = [] def add(self, uid, action, pid, qty=1, price=0): self.events.append({'user':uid, 'action':action, 'product':pid, 'qty':qty, 'price':price}) def state(self, uid): cart = {} for e in self.events: if e['user'] != uid: continue if e['action'] == 'add': cart[e['product']] = {'qty': e['qty'], 'price': e['price']} elif e['action'] == 'remove': cart.pop(e['product'], None) return cart store = CartEventStore() store.add('andi', 'add', 'iPhone-15', 1, 15_000_000) store.add('andi', 'add', 'AirPods', 1, 2_500_000) store.add('andi', 'remove', 'AirPods') # state('andi') → {'iPhone-15': ...} # events stored: 3 (full audit trail!)

Simpan SEMUA event, derive state kapan saja. Replay, audit, time travel — dasar Kafka Streams!

Tips &
Kesalahan Umum

04

Hindari

Kesalahan Umum Stream Processing

  • Proses satu-satu tanpa window -- kehilangan konteks waktu. Fix: Selalu gunakan windowing untuk agregasi.
  • Tidak handle late events -- data telat masuk = hasil salah. Fix: Watermark + allowed lateness.
  • State tidak dibersihkan -- memory terus naik. Fix: TTL (time-to-live) pada state.
  • Tidak idempotent -- retry menyebabkan duplikat. Fix: Dedup key atau exactly-once.
  • Window terlalu kecil -- noise tinggi, alert palsu. Fix: Sesuaikan window size dengan use case.
