Teori &
Konsep

02

Perbandingan

Tradisional vs Cloud Data Warehouse

DB Tradisional

  • CPU + Disk terikat (satu mesin)
  • Beli/sewa server sendiri, mahal
  • Butuh DBA untuk maintenance
  • Bayar 24/7 meskipun idle
  • Scale up = beli hardware baru
  • Kapasitas terbatas

Cloud Data Warehouse

  • Compute & storage TERPISAH
  • Serverless, tidak kelola infra
  • Auto-scale ribuan CPU otomatis
  • Bayar per query (pay-as-you-go)
  • Scale elastis dalam hitungan detik
  • Petabyte-scale storage
◇

Arsitektur

Arsitektur Google BigQuery

1
User / SQL Query
Anda menulis SQL di console, Colab, atau API
2
DREMEL Engine
Distributed SQL engine: ribuan CPU memproses query secara paralel
3
COLOSSUS Storage
Columnar storage terdistribusi: data disimpan per kolom, terkompresi
4
Hasil
Miliaran baris di-scan dalam hitungan detik, hasil dikembalikan ke user

DREMEL + COLOSSUS = kombinasi yang membuat BigQuery bisa query 1 TB dalam < 10 detik.

Konsep Kunci

Separation of Compute & Storage

Coupled (Tradisional)

  • CPU & disk di satu server
  • Tambah storage = beli server baru
  • Tambah CPU = migrasi data
  • Bayar keduanya sekaligus
  • Tidak bisa scale independen

Decoupled (BigQuery)

  • Compute dan storage terpisah
  • Tambah storage tanpa ubah compute
  • Tambah CPU tanpa pindah data
  • Bayar storage murah, compute per query
  • Scale masing-masing independen

Kenapa penting? Simpan 100 TB data (murah), tapi hanya bayar compute saat query dijalankan.

⚙

Optimasi

Partitioning & Clustering

Partitioning

  • Bagi tabel berdasarkan kolom (biasanya tanggal)
  • WHERE date = '2024-01-01' hanya scan 1 partisi
  • Tanpa partisi: scan SEMUA data (mahal!)
  • Hemat biaya hingga 90%

Clustering

  • Urutkan data dalam partisi berdasarkan kolom
  • WHERE kota = 'Jakarta' langsung ke blok relevan
  • Bisa cluster hingga 4 kolom
  • Filter lebih cepat, scan lebih sedikit
-- Buat tabel dengan partisi + cluster CREATE TABLE dataset.trips PARTITION BY DATE(waktu) CLUSTER BY kota, layanan AS SELECT * FROM raw_trips

SQL Lanjut

SQL Tingkat Lanjut di BigQuery

CTE (WITH)

Sub-query bernama untuk query kompleks yang mudah dibaca dan di-debug.

Window Functions

ROW_NUMBER, RANK, LAG, LEAD, Moving Average tanpa mengurangi baris.

UNNEST

Buka array/nested fields menjadi baris. Kunci untuk data GA4, Firebase, JSON.

APPROX Functions

APPROX_COUNT_DISTINCT, APPROX_QUANTILES: 100x cepat, error < 2%.

Perbandingan

BigQuery vs DuckDB — Kapan Pakai Apa?

BigQuery

  • Data > 10 GB atau TB-scale
  • Multi-user / tim kolaborasi
  • Dashboard & BI (Looker, Data Studio)
  • Data sudah ada di GCP
  • Scheduled queries & pipelines
  • 200+ public datasets siap pakai

DuckDB

  • Data < 10 GB, single machine
  • Eksplorasi lokal / notebook
  • Prototyping cepat tanpa cloud
  • File lokal: CSV, Parquet, JSON
  • Zero setup, zero cost
  • Integrasi Pandas/Polars

Pattern terbaik: Eksplorasi kecil di DuckDB, lalu scale ke BigQuery untuk produksi.

Praktikum

03

Praktikum 1

Query NYC Taxi — Puluhan Juta Baris

-- Query public dataset NYC Taxi (puluhan juta baris) SELECT EXTRACT(HOUR FROM pickup_datetime) AS jam, COUNT(*) AS total_trips, ROUND(AVG(trip_distance), 2) AS avg_jarak, ROUND(AVG(total_amount), 2) AS avg_tarif FROM `bigquery-public-data.new_york_taxi_trips.tlc_yellow_trips_2022` WHERE pickup_datetime BETWEEN '2022-01-01' AND '2022-03-31' GROUP BY jam ORDER BY jam

Puluhan juta baris dalam 3-5 detik! Perhatikan estimasi GB di kanan atas sebelum run.

Praktikum 2

UNNEST: Nested & Repeated Fields

Data BigQuery sering punya kolom ARRAY atau STRUCT (nested). UNNEST membuka array menjadi baris.

-- Contoh: GA4 events (nested data) SELECT event_name, params.key AS param_key, params.value.string_value AS param_value FROM `bigquery-public-data.ga4_obfuscated_sample_ecommerce.events_*`, UNNEST(event_params) AS params WHERE _TABLE_SUFFIX BETWEEN '20210101' AND '20210131' AND event_name = 'page_view' LIMIT 100

UNNEST = kunci untuk data GA4, Firebase, dan JSON nested. Wajib dikuasai!

Praktikum 3

Window Functions di BigQuery

-- Trend harian + ranking per bulan WITH harian AS ( SELECT DATE(pickup_datetime) AS tgl, COUNT(*) AS trips, ROUND(AVG(total_amount), 2) AS avg_tarif FROM `bigquery-public-data.new_york_taxi_trips.tlc_yellow_trips_2022` WHERE pickup_datetime BETWEEN '2022-01-01' AND '2022-01-31' GROUP BY tgl ) SELECT tgl, trips, avg_tarif, AVG(trips) OVER (ORDER BY tgl ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS ma_7d, RANK() OVER (ORDER BY trips DESC) AS rank_trips FROM harian ORDER BY tgl

Praktikum 4

APPROX Functions — Untuk Data Masif

-- Perbandingan: exact vs approximate -- LAMBAT pada TB data (exact count) SELECT COUNT(DISTINCT pickup_location_id) AS exact_count FROM `bigquery-public-data.new_york_taxi_trips.tlc_yellow_trips_2022` -- CEPAT: approximate (error < 2%) SELECT APPROX_COUNT_DISTINCT(pickup_location_id) AS approx_count FROM `bigquery-public-data.new_york_taxi_trips.tlc_yellow_trips_2022` -- Approximate percentiles SELECT APPROX_QUANTILES(total_amount, 100)[OFFSET(50)] AS median_tarif, APPROX_QUANTILES(total_amount, 100)[OFFSET(95)] AS p95_tarif FROM `bigquery-public-data.new_york_taxi_trips.tlc_yellow_trips_2022`

Gunakan APPROX untuk data TB-scale. Hasil hampir sama, tapi 10-100x lebih cepat!

Praktikum 5

Pattern: BigQuery + DuckDB

Ambil data besar dari BigQuery, lalu analisis lokal dengan DuckDB. Best of both worlds.

# Step 1: Query BigQuery → simpan ke DataFrame query = """ SELECT DATE(pickup_datetime) AS tgl, COUNT(*) AS trips, AVG(total_amount) AS avg_tarif FROM `bigquery-public-data.new_york_taxi_trips.tlc_yellow_trips_2022` WHERE pickup_datetime BETWEEN '2022-01-01' AND '2022-06-30' GROUP BY tgl """ df_bq = client.query(query).to_dataframe() # Step 2: Simpan ke Parquet df_bq.to_parquet('nyc_daily.parquet') # Step 3: Analisis lokal dengan DuckDB import duckdb duckdb.sql(""" SELECT tgl, trips, avg_tarif, AVG(trips) OVER (ORDER BY tgl ROWS 6 PRECEDING) AS ma_7d FROM read_parquet('nyc_daily.parquet') ORDER BY tgl """).show()

Bagian 4

Tips &
Kesalahan Umum

04

Hindari

Kesalahan Umum BigQuery

  • SELECT * pada tabel besar → kuota 1 TB cepat habis! Fix: SELECT hanya kolom yang dibutuhkan
  • Query tanpa WHERE tanggal pada tabel partisi → scan semua partisi (mahal). Fix: Selalu filter tanggal
  • Tidak cek estimasi GB sebelum run → kaget kuota habis. Fix: Lihat validator di kanan atas console
  • COUNT(DISTINCT x) pada data TB → lambat & mahal. Fix: Pakai APPROX_COUNT_DISTINCT
  • Lupa LIMIT saat eksplorasi → download jutaan baris ke browser. Fix: Selalu LIMIT 100 saat eksplorasi
  • Ignore nested data → query error pada kolom ARRAY. Fix: Pakai UNNEST untuk buka nested fields
