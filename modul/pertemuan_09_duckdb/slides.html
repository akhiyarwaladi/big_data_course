Konsep Inti

Apa Itu DuckDB?

Database analitik in-process — jalan DI DALAM Python. Seperti SQLite untuk transaksi, DuckDB untuk analytics.

Columnar Storage

Hanya baca kolom yang di-SELECT

Multi-threaded

Otomatis pakai SEMUA CPU core

Query File Langsung

CSV, Parquet, JSON tanpa import

SQL Standard

GROUP BY, Window, CTE, QUALIFY

Arsitektur

OLAP vs OLTP

OLTP (Transactional)

  • INSERT, UPDATE, DELETE per baris
  • Aplikasi: e-commerce, perbankan
  • Row-based storage
  • Contoh: MySQL, PostgreSQL
  • Ratusan ribu transaksi/detik

OLAP (Analytical)

  • SELECT, GROUP BY, JOIN massal
  • Analisis: dashboard, reporting
  • Columnar storage
  • Contoh: DuckDB, BigQuery
  • Miliar baris dalam detik

Storage

Columnar vs Row Storage

Row-based (CSV, MySQL)

  • Data disimpan per BARIS
  • [Andi, Jakarta, 25000]
  • [Budi, Bandung, 18000]
  • SELECT AVG(tarif) → Baca SEMUA kolom!
  • Buang-buang I/O

Columnar (Parquet, DuckDB)

  • Data disimpan per KOLOM
  • nama: [Andi, Budi]
  • tarif: [25000, 18000]
  • SELECT AVG(tarif) → Baca tarif SAJA!
  • 5-10x lebih cepat

Format Data

Parquet vs CSV

AspekCSVParquet
FormatText, human-readableBinary, columnar
KompresiTidak adaSnappy/Zstd (otomatis)
Tipe DataTidak ada (semua string)Typed (int, float, date)
Ukuran (10M rows)~800 MB~120 MB (6.7x kecil)
Read+Query3.2 detik0.3 detik (10x cepat)

Parquet = standar industri. Dipakai BigQuery, Snowflake, Databricks, Spark, AWS Athena.

Performa

Vectorized Execution & Optimasi

Vectorized Processing

Proses data dalam batch (vektor), bukan satu per satu. Memanfaatkan SIMD CPU modern.

Predicate Pushdown

Filter diterapkan SEBELUM baca data. Hanya baca data yang lolos filter.

Projection Pushdown

Hanya baca KOLOM yang dibutuhkan. SELECT nama → skip kolom lain.

Parallel Execution

Otomatis gunakan semua CPU core untuk query besar.

Praktikum

Praktikum 1

Generate 10 Juta Baris GoRide

CREATE TABLE trips AS WITH rng AS ( SELECT row_number() OVER() AS rn, random() AS r_kota, random() AS r_layanan, random() AS r_tarif, random() AS r_tarif_val, random() AS r_status, random() AS r_waktu1, random() AS r_waktu2, random() AS r_rating FROM generate_series(1, 10000000) ) SELECT 'TRIP-' || LPAD(rn::VARCHAR, 9, '0') AS trip_id, TIMESTAMP '2025-01-01' + INTERVAL (r_waktu1*90) DAY + INTERVAL (r_waktu2*24) HOUR AS waktu_pesan, CASE WHEN r_kota < 0.28 THEN 'Jakarta' WHEN r_kota < 0.43 THEN 'Surabaya' WHEN r_kota < 0.55 THEN 'Bandung' WHEN r_kota < 0.64 THEN 'Medan' ELSE 'Semarang' END AS kota, CASE WHEN r_layanan < 0.35 THEN 'GoRide' WHEN r_layanan < 0.58 THEN 'GoCar' WHEN r_layanan < 0.78 THEN 'GoFood' ELSE 'GoSend' END AS layanan, ROUND(CASE WHEN r_tarif < 0.35 THEN 7000 + r_tarif_val * 43000 ELSE 15000 + r_tarif_val * 85000 END, -2) AS tarif, CASE WHEN r_rating < 0.14 THEN 3 WHEN r_rating < 0.38 THEN 4 ELSE 5 END AS rating, CASE WHEN r_status < 0.84 THEN 'completed' ELSE 'cancelled' END AS status FROM rng

10 juta baris dalam ~3 detik! CTE + random() per kolom → distribusi akurat. Tanpa CTE, distribusi miring!

SQL

GROUP BY: Performa per Kota

SELECT kota, COUNT(*) AS total_trip, ROUND(AVG(tarif)) AS avg_tarif, ROUND(SUM(tarif) / 1e9, 2) AS revenue_miliar, ROUND(AVG(rating), 2) AS avg_rating, ROUND(AVG(jarak_km), 1) AS avg_jarak, ROUND(COUNT(*) FILTER (WHERE status = 'completed') * 100.0 / COUNT(*), 1) AS completion_pct FROM trips GROUP BY kota ORDER BY revenue_miliar DESC

10M baris < 0.1 detik! Insight: Jakarta revenue tertinggi (~28%), FILTER clause hitung completion rate tanpa subquery.

SQL

Window Functions: Trend Harian

WITH harian AS ( SELECT waktu_pesan::DATE AS tgl, COUNT(*) AS trips, ROUND(SUM(tarif) / 1e6, 1) AS revenue_juta FROM trips WHERE status = 'completed' GROUP BY tgl ) SELECT tgl, trips, revenue_juta, -- Moving Average 7 hari ROUND(AVG(revenue_juta) OVER ( ORDER BY tgl ROWS BETWEEN 6 PRECEDING AND CURRENT ROW), 1) AS ma7, -- Delta dari hari sebelumnya ROUND(revenue_juta - LAG(revenue_juta) OVER (ORDER BY tgl), 1) AS delta, -- Ranking revenue RANK() OVER (ORDER BY revenue_juta DESC) AS rank_rev FROM harian ORDER BY tgl LIMIT 20

SQL

CTE: Common Table Expressions

CTE = "sub-query bernama" yang membuat SQL lebih mudah dibaca dan di-debug.

WITH kota_stats AS ( SELECT kota, COUNT(*) AS trips, SUM(tarif) AS total_rev FROM trips GROUP BY kota ), ranked AS ( SELECT *, RANK() OVER (ORDER BY total_rev DESC) AS rnk FROM kota_stats ) SELECT * FROM ranked WHERE rnk <= 5

Fitur

Query File Langsung — Tanpa Import!

-- Query CSV langsung SELECT * FROM read_csv('data.csv') LIMIT 10 -- Query Parquet langsung SELECT kota, COUNT(*) FROM read_parquet('trips.parquet') GROUP BY kota -- Query JSON SELECT * FROM read_json('events.json') -- Export ke Parquet COPY trips TO 'trips.parquet' (FORMAT PARQUET)

Integrasi

DuckDB + Python: Zero-Copy

import pandas as pd import duckdb # Buat DataFrame Pandas df = pd.DataFrame({'nama': ['Andi','Budi'], 'kota': ['JKT','BDG']}) # Query DataFrame pakai SQL — zero copy! result = duckdb.sql("SELECT kota, COUNT(*) FROM df GROUP BY kota") # Konversi balik result.df() # → Pandas result.pl() # → Polars result.arrow() # → Arrow

SQL

Pola Jam Sibuk & Revenue per Layanan

Jam Sibuk

SELECT EXTRACT(HOUR FROM waktu_pesan) AS jam, COUNT(*) AS trips, ROUND(AVG(tarif)) AS avg_tarif, REPEAT('█', GREATEST( (COUNT(*) / 8000)::INT, 1) ) AS volume FROM trips WHERE status = 'completed' GROUP BY jam ORDER BY jam

Revenue per Layanan

SELECT layanan, COUNT(*) AS trips, ROUND(SUM(tarif)/1e9,2) AS rev_miliar, ROUND(AVG(rating), 2) AS avg_rating, ROUND(COUNT(*) FILTER ( WHERE pakai_promo ) * 100.0 / COUNT(*), 1) AS promo_pct FROM trips WHERE status='completed' GROUP BY layanan ORDER BY rev_miliar DESC

SQL

Adopsi Pembayaran Digital vs Cash

SELECT metode_bayar, COUNT(*) AS transaksi, ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER(), 1) AS persen, ROUND(SUM(tarif) / 1e9, 2) AS rev_miliar, REPEAT('█', (COUNT(*) * 40 / SUM(COUNT(*)) OVER())::INT) AS bar FROM trips WHERE status = 'completed' GROUP BY metode_bayar ORDER BY transaksi DESC

Window Function dalam GROUP BY! SUM(COUNT(*)) OVER() hitung total tanpa subquery. REPEAT() buat bar chart langsung di SQL.

Analisis

Kenapa DuckDB Lebih Cepat?

1. Columnar

Skip kolom tidak diperlukan. Hanya baca data yang di-SELECT.

2. Multi-threaded

Semua CPU core bekerja paralel secara otomatis.

3. Vectorized

Proses batch 1024+ values per operasi, bukan satu-satu.

4. C++ Engine

Native code, tanpa overhead JVM atau Python interpreter.

Kesalahan Umum

04

Hindari

Kesalahan Umum

  • SELECT * pada data besar → lambat & boros memory. Fix: SELECT hanya kolom yang dibutuhkan
  • Data besar di CSV → lambat & boros disk. Fix: Convert ke Parquet (COPY TO ... FORMAT PARQUET)
  • Pandas untuk > 5M baris → lambat atau OOM. Fix: DuckDB (SQL) atau Polars (Python)
  • random() tanpa CTE → distribusi data miring. Fix: Buat CTE dengan random() per kolom

Ringkasan

Apa yang Sudah Kita Pelajari

  • DuckDB = database analitik modern, zero setup, 20–50x lebih cepat dari Pandas
  • Columnar storage = baca hanya kolom yang dibutuhkan
  • Parquet = standar industri (7x lebih kecil, 10x lebih cepat vs CSV)
  • SQL modern = GROUP BY, Window Functions, CTE, FILTER
  • Query file langsung = CSV/Parquet/JSON tanpa import
  • Python integration = zero-copy query Pandas/Polars DataFrames
