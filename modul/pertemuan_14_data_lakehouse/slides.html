Teori &
Konsep

02

Sejarah

3 Era Penyimpanan Data

1990–2010
Era Data Warehouse
Rapi, ACID, SQL-friendly. Tapi mahal (Oracle, Teradata), hanya structured data, sulit scale.
2010–2020
Era Data Lake
Murah (HDFS/S3), semua format (CSV, JSON, gambar). Tapi "data swamp" — no ACID, no schema, no versioning.
2020–Now
Era Lakehouse
Murah + ACID + time travel + schema evolution. Open table formats di atas object storage.
!

Masalah

Masalah Data Lake: "Data Swamp"

No Schema Enforcement

Siapa saja bisa taruh file apapun. Data inconsistent, kolom berubah tanpa kontrol.

No ACID

Partial writes saat crash = data corrupt. Tidak ada rollback. Concurrent writes berantakan.

No Versioning

Tidak bisa query data minggu lalu. Audit trail tidak ada. Salah overwrite = hilang selamanya.

Data Quality Issues

Duplikat, null tanpa validasi, tipe data campur. Siapa yang bertanggung jawab?

80% Data Lake projects gagal karena masalah-masalah di atas (Gartner).

◇

Konsep Inti

Apa Itu Data Lakehouse?

Arsitektur yang menggabungkan keunggulan Data Warehouse (ACID, schema, SQL) dengan keunggulan Data Lake (murah, open format, semua tipe data) menggunakan open table formats di atas object storage.

Dari Warehouse

  • ACID transactions
  • Schema enforcement
  • SQL query engine
  • Data governance

Dari Data Lake

  • Open file formats (Parquet)
  • Murah (object storage)
  • Semua tipe data
  • Decoupled compute/storage

A

Fitur Lakehouse

ACID Transactions

Atomicity

Semua atau tidak sama sekali. Write gagal di tengah? Rollback otomatis, data tetap konsisten.

Consistency

Schema enforced. Kolom harus sesuai tipe data. Tidak bisa taruh string di kolom integer.

Isolation

Multiple writers tidak saling ganggu. Concurrent reads tetap dapat snapshot yang konsisten.

Durability

Sekali committed, data aman. No partial writes. Metadata + data files terpisah tapi sinkron.

Tanpa ACID: partial write saat crash = data corrupt yang sulit di-debug.

T

Fitur Lakehouse

Time Travel

Query data pada versi atau tanggal tertentu di masa lalu. Setiap write membuat snapshot baru, snapshot lama tetap tersedia.

1
Audit Trail
Siapa mengubah apa, kapan? Compliance regulasi jadi mudah.
2
Rollback
Pipeline error? Rollback ke versi sebelumnya dalam hitungan detik.
3
Reprodusibilitas
ML model training ulang pakai data persis sama seperti 3 bulan lalu.
4
Diff & Debug
Bandingkan data versi A vs B. Cari tahu kapan bug masuk pipeline.
S

Fitur Lakehouse

Schema Evolution

Ubah schema tabel tanpa rewrite semua data. Tambah kolom, rename, reorder — backward compatible.

Tanpa Schema Evolution

  • Tambah kolom = rewrite seluruh tabel
  • Downtime berjam-jam untuk ALTER TABLE
  • Pipeline lama break karena schema berubah
  • Biaya compute tinggi untuk migrasi

Dengan Schema Evolution

  • Tambah kolom = update metadata saja
  • Zero downtime, instan
  • Old readers tetap bisa baca (backward compatible)
  • Kolom baru default NULL untuk data lama

Arsitektur

Medallion Architecture

B
Bronze Layer (Raw)
Data mentah dari sumber. Tidak diubah. Append-only. Parquet partitioned by date/month. "Single source of truth" mentah.
S
Silver Layer (Cleaned)
Deduplikasi, validasi tipe data, filter invalid rows, tambah derived columns. Schema enforced. Siap untuk analisis.
G
Gold Layer (Marts)
Agregasi bisnis: daily_revenue, sla_compliance, customer_metrics. Siap untuk dashboard dan reporting.

Partitioning di Bronze memungkinkan partition pruning — query hanya baca bulan yang dibutuhkan.

B

Praktikum 1

Build Bronze Layer

# CSV → Partitioned Parquet by month import polars as pl df = pl.DataFrame(rows) # Partitioned write: satu file per bulan for month, group in df.group_by('month'): path = f'bronze/shipments_{month[0]}.parquet' group.write_parquet(path) print(f' Wrote {len(group)} rows → {path}') # Simpan metadata versi version_info = { 'version': 1, 'timestamp': datetime.datetime.now().isoformat(), 'row_count': len(df), 'columns': df.columns, } json.dump([version_info], open('metadata/versions.json','w'))

Partitioning by month = query "WHERE month = '2024-06'" hanya baca 1 file, bukan 12.

S

Praktikum 2

Build Silver Layer

# Read all bronze parquet files bronze = pl.read_parquet('bronze/*.parquet') # Clean & validate silver = (bronze .filter(pl.col('weight_kg') > 0) .filter(pl.col('cost') > 0) .filter(pl.col('origin') != pl.col('destination')) .with_columns([ (pl.col('cost') / pl.col('weight_kg')) .round(2).alias('cost_per_kg'), (pl.col('actual_days') <= pl.col('sla_days')) .alias('sla_met'), ]) .unique(subset=['order_id']) ) silver.write_parquet('silver/shipments_clean.parquet') print(f'Bronze: {len(bronze)} → Silver: {len(silver)}')
TT

Praktikum 3

Time Travel Simulation

# Simulasi multiple versions (snapshot per write) versions = json.load(open('metadata/versions.json')) # Version 2: tambah data baru (append) new_rows = [{'order_id': f'ORD-{50000+i:06d}', 'kota': random.choice(['Jakarta','Surabaya']), 'tarif': random.randint(10000,90000)} for i in range(5000)] df_new = pl.DataFrame(new_rows) df_new.write_parquet('bronze/shipments_2024-13_patch.parquet') versions.append({ 'version': 2, 'timestamp': datetime.datetime.now().isoformat(), 'row_count': 55000, 'action': 'append 5000 rows' }) json.dump(versions, open('metadata/versions.json','w')) # Read old snapshot (version 1) v1 = [v for v in versions if v['version'] == 1][0] print(f"Version 1 had {v1['row_count']} rows at {v1['timestamp']}")

Praktikum 4

Schema Evolution Simulation

# Evolusi: tambah kolom tax tanpa rewrite data lama silver = pl.read_parquet('silver/shipments_clean.parquet') # Schema v2: tambah kolom tax (11%) silver_v2 = silver.with_columns([ (pl.col('cost') * 0.11).round(0).cast(pl.Int64).alias('tax'), (pl.col('cost') * 1.11).round(0).cast(pl.Int64).alias('total_with_tax'), ]) silver_v2.write_parquet('silver/shipments_clean_v2.parquet') # Backward compatible: data v1 tetap bisa dibaca old_data = pl.read_parquet('silver/shipments_clean.parquet') print(f"Old schema: {old_data.columns}") print(f"New schema: {silver_v2.columns}") # Update schema metadata schema_info = {'version': 2, 'columns': silver_v2.columns, 'added': ['tax', 'total_with_tax']} json.dump(schema_info, open('metadata/schema.json','w'))
G

Praktikum 5

Build Gold Layer — Aggregated Marts

# Gold Mart 1: Daily Revenue silver = pl.read_parquet('silver/shipments_clean_v2.parquet') daily_revenue = (silver .group_by('ship_date') .agg([ pl.col('cost').sum().alias('total_revenue'), pl.col('order_id').count().alias('order_count'), pl.col('cost').mean().round(0).alias('avg_order_value'), ]) .sort('ship_date') ) daily_revenue.write_parquet('gold/daily_revenue.parquet') # Gold Mart 2: SLA Compliance per Route sla_report = (silver .group_by(['origin', 'destination']) .agg([ pl.col('sla_met').mean().round(3).alias('sla_rate'), pl.col('order_id').count().alias('total_orders'), ]) ) sla_report.write_parquet('gold/sla_compliance.parquet')

Praktikum 5b

Query Gold Layer dengan DuckDB

# DuckDB langsung query Parquet tanpa import! con = duckdb.connect() # Top 10 hari revenue tertinggi con.sql(""" SELECT ship_date, total_revenue, order_count, avg_order_value FROM 'gold/daily_revenue.parquet' ORDER BY total_revenue DESC LIMIT 10 """).show() # Rute dengan SLA terburuk con.sql(""" SELECT origin, destination, sla_rate, total_orders FROM 'gold/sla_compliance.parquet' WHERE total_orders > 100 ORDER BY sla_rate ASC LIMIT 5 """).show()

DuckDB + Parquet = query analitik tanpa database server. Sempurna untuk Lakehouse lokal.

Praktikum 6

Data Quality Checks

# Automated quality checks — simpan ke metadata silver = pl.read_parquet('silver/shipments_clean_v2.parquet') checks = { 'total_rows': len(silver), 'null_counts': {col: silver[col].null_count() for col in silver.columns}, 'unique_orders': silver['order_id'].n_unique(), 'duplicate_check': len(silver) == silver['order_id'].n_unique(), 'min_cost': silver['cost'].min(), 'max_cost': silver['cost'].max(), 'sla_rate_overall': round(silver['sla_met'].mean(), 3), } json.dump(checks, open('metadata/quality.json','w'), default=str) for k, v in checks.items(): print(f" {k}: {v}")

Production tip: jalankan quality checks setelah setiap ETL step. Gagal? Jangan lanjut ke layer berikutnya!

Tips &
Kesalahan Umum

04

Tips

Tips Membangun Lakehouse

  • Selalu mulai dari Bronze: simpan data mentah tanpa transformasi. Jangan pernah buang raw data.
  • Partition dengan bijak: partition by date/month untuk time-series. Jangan over-partition (terlalu banyak file kecil).
  • Schema versioning: simpan schema di metadata. Track setiap perubahan seperti git untuk data.
  • Idempotent pipelines: jalankan ulang pipeline tanpa duplikasi data. Gunakan UPSERT/MERGE.
  • Quality gates: validasi di setiap layer. Bronze-to-Silver paling kritis.
  • Gunakan Parquet, bukan CSV: kompresi 10x lebih kecil, columnar pruning, tipe data preserved.
!

Kesalahan Umum

Kesalahan Umum & Solusinya

Kesalahan

  • Langsung transform di Bronze
  • Tidak ada metadata/versioning
  • Partition by high-cardinality column
  • Skip quality checks
  • Gold layer terlalu banyak kolom
  • Campur raw + cleaned di satu folder

Solusi

  • Bronze = raw only, transform di Silver
  • Selalu track version + schema di metadata/
  • Partition by month/day, bukan by user_id
  • Quality gate wajib sebelum naik layer
  • Gold = minimal kolom, spesifik per use case
  • Pisahkan folder: bronze/, silver/, gold/

Ringkasan

Ringkasan Pertemuan 14

Lakehouse = Lake + Warehouse

Murah seperti data lake, rapi seperti warehouse. Open table formats di atas object storage.

ACID + Time Travel

No partial writes, query data versi lama, rollback kapan saja. Data engineering yang aman.

Schema Evolution

Tambah/ubah kolom tanpa rewrite data. Backward compatible. Zero downtime migration.

Medallion: B → S → G

Bronze (raw) → Silver (clean) → Gold (marts). Partitioning + quality gates di setiap layer.

Iceberg vs Delta vs Hudi

Iceberg paling populer (engine-agnostic). Delta kuat di Databricks. Hudi untuk streaming/upsert.

DuckDB + Parquet

Query langsung di atas Parquet tanpa server. Perfect untuk lakehouse lokal dan pembelajaran.
