Tips

Tips Teknis ChromaDB di Colab

  • get_or_create_collection bukan create_collection → aman untuk re-run tanpa error
  • Data HILANG saat disconnect (in-memory) — gunakan PersistentClient untuk Google Drive
  • pip install tokenizers --upgrade jika muncul error tokenizer
  • Model all-MiniLM-L6-v2 — multilingual, gratis, ~80MB, 384 dimensi
  • Jangan idle > 90 menit — auto disconnect, data in-memory hilang
  • Mount Drive untuk persistent storage: chromadb.PersistentClient(path="/content/drive/...")

Konsep Inti

RAG: Retrieve → Augment → Generate

1
User Question
"Berapa ongkir ke Surabaya?"
2
Retrieve
Cari dokumen relevan di Vector DB (semantic search)
3
Augment
Tambahkan konteks dokumen ke prompt LLM
4
Generate
LLM jawab akurat berdasarkan data → "Ongkir ke Surabaya Rp 15.000"

RAG = Retrieval-Augmented Generation. LLM + database Anda = AI yang akurat dan up-to-date.

Konsep

Embeddings: Representasi Numerik Makna

Embedding = mengubah teks menjadi vektor angka (384 dimensi) yang menangkap MAKNA, bukan hanya kata.

Mirip = Dekat

"kucing lucu" ≈ "cat cute" — beda bahasa, vektor dekat karena makna sama

Beda = Jauh

"kucing lucu" ≠ "mobil cepat" — vektor jauh karena makna berbeda

384 Dimensi

Model all-MiniLM-L6-v2 menghasilkan vektor [0.023, -0.156, 0.891, ...] dengan 384 angka

Semantic Search

Cari berdasarkan MAKNA, bukan keyword exact match seperti SQL LIKE

Konsep

Similarity Search: Mengukur Kedekatan

MetodeCara KerjaRangeKapan Dipakai
Cosine SimilaritySudut antar vektor-1 s/d 1Paling umum, default ChromaDB
Dot ProductPerkalian titik vektor-∞ s/d +∞Cepat, untuk normalized vectors
Euclidean DistanceJarak geometris0 s/d +∞Ketika magnitudo penting
K-Nearest NeighborsCari K terdekatTop-K resultsRetrieval: ambil top 3-5 dokumen

ChromaDB default = cosine similarity. Semakin dekat ke 1.0, semakin mirip maknanya.

Strategi

Chunking: Memecah Dokumen

Dokumen besar harus dipecah menjadi chunk kecil agar embedding lebih presisi. Max ~500 karakter per chunk.

Fixed-size

Potong setiap N karakter. Simpel tapi bisa potong di tengah kalimat.

Sentence-based

Potong per kalimat/paragraf. Menjaga konteks utuh.

Semantic

Potong berdasarkan perubahan topik. Paling akurat, paling kompleks.

Overlap

Chunk saling tumpang tindih 50-100 char. Mencegah kehilangan konteks di batas.

⚙

Arsitektur

RAG Pipeline Architecture

Indexing (Offline)

  • Load dokumen (PDF, CSV, TXT)
  • Chunk pecah jadi potongan kecil
  • Embed ubah chunk jadi vektor
  • Store simpan di Vector DB

Query (Online)

  • Embed query ubah pertanyaan jadi vektor
  • Retrieve top-K cari chunk terdekat
  • Augment prompt gabung konteks + pertanyaan
  • Generate LLM jawab dengan konteks

Perbandingan

RAG vs Fine-tuning

RAG

  • Data bisa diupdate kapan saja
  • Tidak perlu training ulang
  • Murah — hanya butuh Vector DB
  • Jawaban bisa di-trace ke sumber
  • Cocok: FAQ, docs, knowledge base

Fine-tuning

  • Data "baked in" ke model
  • Butuh GPU mahal + waktu lama
  • Biaya $100-$10.000+ per training
  • Sulit trace sumber jawaban
  • Cocok: style/tone, domain khusus

80% use case cukup RAG! Fine-tuning hanya jika butuh ubah perilaku/gaya model.

Eksplorasi Embeddings: Similarity Matrix

from sentence_transformers import SentenceTransformer from numpy.linalg import norm import numpy as np model = SentenceTransformer('all-MiniLM-L6-v2') # 384 dim, gratis texts = [ "Harga iPhone 15 Pro Max terbaru", "Berapa biaya Apple smartphone flagship", "Cara memasak nasi goreng yang enak", "Resep masakan Indonesia pedas", "Jadwal kereta api Jakarta Surabaya", "Transportasi kereta ibukota ke Jawa Timur", ] emb = model.encode(texts) # shape: (6, 384) def cos_sim(a, b): return np.dot(a, b) / (norm(a) * norm(b)) # T1≈T2 (iPhone/smartphone): 0.85 # T3≈T4 (masakan): 0.78 # T5≈T6 (kereta): 0.82 # T1↔T3 (unrelated): 0.12

Teks beda bahasa/kata tapi makna sama → similarity tinggi! Ini kekuatan semantic search.

Semantic Search: Cari Berdasarkan Makna

# Query semantic — bukan keyword match! results = collection.query( query_texts=["berapa biaya kirim ke Jawa Timur?"], n_results=2 ) # Hasil: dokumen paling relevan for doc, dist in zip(results['documents'][0], results['distances'][0]): print(f"[{dist:.3f}] {doc}") # Output: # [0.412] Ongkir ke Jakarta Rp 10.000, Surabaya Rp 15.000 # [1.234] Retur bisa dilakukan dalam 7 hari...

"biaya kirim ke Jawa Timur" menemukan dokumen "ongkir ke Surabaya" — tanpa keyword match!

Praktikum 3

RAG Pipeline: Konteks + LLM

def rag_answer(question, collection, n_results=3): # Step 1: Retrieve dokumen relevan results = collection.query( query_texts=[question], n_results=n_results ) context = "\n".join(results['documents'][0]) # Step 2: Augment — buat prompt dengan konteks prompt = f"""Berdasarkan informasi berikut: {context} Jawab pertanyaan: {question} Jawab hanya berdasarkan informasi di atas.""" # Step 3: Generate — kirim ke LLM (atau jawab manual) print("KONTEKS:", context) print("PROMPT:", prompt) return prompt rag_answer("Bagaimana cara retur barang?", collection)

Kesalahan Umum

04

Hindari

Kesalahan Umum RAG & ChromaDB

  • create_collection saat re-run → ERROR duplikat. Fix: gunakan get_or_create_collection
  • Chunk terlalu besar (> 1000 char) → embedding kurang presisi. Fix: maks ~500 char per chunk
  • Threshold relevance terlalu rendah → ambil dokumen tidak relevan. Fix: filter distance < 1.0
  • Tidak pakai metadata → pencarian tidak fokus. Fix: tambahkan category, source, date
  • ID duplikat saat add → data tertimpa diam-diam. Fix: gunakan ID unik (uuid atau prefix)
  • Lupa TPU crash → ChromaDB/sentence-transformers error di TPU. Fix: pakai CPU atau GPU
