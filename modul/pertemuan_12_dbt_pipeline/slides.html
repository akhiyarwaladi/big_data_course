Perbandingan

ETL vs ELT

ETL (Tradisional)

  • Transform di server TERPISAH
  • Infra khusus: Informatica, SSIS, Talend
  • Bahasa: Java, Scala, custom scripts
  • Mahal (lisensi + server)
  • Testing manual, sulit di-maintain
  • Tidak ada version control

ELT (Modern)

  • Transform DI DALAM warehouse
  • SQL biasa — semua orang bisa
  • dbt: gratis & open source
  • Git, testing, docs built-in
  • 40,000+ perusahaan pakai dbt
  • CI/CD untuk data pipeline

ELT menang karena warehouse modern sudah sangat powerful — kenapa transform di luar?

dbt

Konsep Inti

Apa Itu dbt?

dbt = data build tool. Framework transformasi data berbasis SQL. Setiap model = satu file .sql berisi satu SELECT statement.

SQL-First

Tulis SELECT, dbt yang handle CREATE TABLE / INSERT

Version Control

Semua model di Git — review, rollback, kolaborasi

Testing Built-in

not_null, unique, accepted_values — otomatis

Documentation

Auto-generate docs + lineage DAG visual

Fitur

dbt: Fitur Utama

Models (.sql)

Setiap file .sql = satu transformasi. SELECT saja, dbt handle sisanya.

ref() Dependency

ref('model_name') — dbt otomatis bangun urutan eksekusi (DAG).

Tests

not_null, unique, accepted_values, relationships — validasi otomatis.

Docs + Lineage

dbt docs generate → website interaktif dengan DAG visual.

Materializations

view, table, incremental, ephemeral — pilih sesuai kebutuhan.

Macros (Jinja)

Template SQL reusable — DRY principle untuk data pipeline.

Arsitektur

Arsitektur Berlapis (Medallion)

1
Sources
Data mentah dari database / API / file. Didefinisikan di sources.yml.
2
Staging (stg_)
1:1 dengan source. Cleaning, rename, type casting. Tidak ada JOIN.
3
Intermediate (int_)
JOIN antar staging models. Enrichment, kalkulasi bisnis.
4
Marts (fct_ / dim_)
Siap konsumsi! Fact tables + dimension tables untuk dashboard.

Aturan emas: source() hanya di staging, ref() di semua layer setelahnya.

Konsep

ref() dan source() — Dependency Graph

source()

-- stg_orders.sql SELECT * FROM {{ source('raw', 'raw_orders') }}

Referensi ke tabel sumber (raw data). Didefinisikan di sources.yml.

ref()

-- int_order_items.sql SELECT * FROM {{ ref('stg_orders') }} o JOIN {{ ref('stg_products') }} p ON o.product_id = p.product_id

Referensi antar model. dbt bangun DAG otomatis dari ref().

DAG = Directed Acyclic Graph. dbt tahu model mana harus dijalankan duluan berdasarkan ref().

Quality

Testing Framework dbt

Built-in Tests (schema.yml)

  • not_null — kolom tidak boleh NULL
  • unique — tidak ada duplikat
  • accepted_values — hanya nilai tertentu
  • relationships — foreign key valid

Custom SQL Tests

  • File .sql di folder tests/
  • Query yang HARUS return 0 rows
  • Contoh: order_date tidak di masa depan
  • Contoh: total revenue > 0
-- tests/assert_no_future_orders.sql SELECT order_id, order_date FROM {{ ref('stg_orders') }} WHERE order_date > current_date()

Materialization Strategies

TipeCara KerjaKapan Pakai
viewCREATE VIEW — tidak simpan dataModel ringan, staging layer
tableCREATE TABLE — rebuild penuhMart tables, data kecil-sedang
incrementalINSERT baris baru sajaData besar, append-only (log, events)

-- Contoh konfigurasi di model {{ config(materialized='incremental', unique_key='order_id') }} SELECT * FROM {{ source('raw', 'raw_orders') }} {% if is_incremental() %} WHERE order_date > (SELECT MAX(order_date) FROM {{ this }}) {% endif %}

Bagian 3

Praktikum

03

Praktikum 1

Staging Model: stg_orders.sql

-- models/staging/stg_orders.sql {{ config(materialized='view') }} WITH source AS ( SELECT * FROM {{ source('raw', 'raw_orders') }} ), cleaned AS ( SELECT order_id, customer_id, product_id, COALESCE(quantity, 1) AS quantity, order_date, CASE WHEN status IN ('completed','pending','cancelled') THEN status ELSE 'unknown' END AS order_status FROM source ) SELECT * FROM cleaned

Staging = 1:1 dengan source. Hanya cleaning, rename, COALESCE. Tidak ada JOIN.

Praktikum 2

Intermediate: int_order_items.sql

-- models/intermediate/int_order_items.sql {{ config(materialized='view') }} WITH orders AS ( SELECT * FROM {{ ref('stg_orders') }} ), products AS ( SELECT * FROM {{ ref('stg_products') }} ) SELECT o.order_id, o.customer_id, o.order_date, o.order_status, o.quantity, p.product_name, p.category, p.price, o.quantity * p.price AS line_total FROM orders o JOIN products p ON o.product_id = p.product_id

Intermediate = tempat JOIN. Gunakan ref() untuk merujuk staging models.

Praktikum 3

Mart Models: fct_sales & dim_customers

fct_sales.sql

-- models/marts/fct_sales.sql {{ config(materialized='table') }} SELECT order_id, customer_id, order_date, COUNT(*) AS items, SUM(line_total) AS revenue FROM {{ ref('int_order_items') }} WHERE order_status = 'completed' GROUP BY 1,2,3

dim_customers.sql

-- models/marts/dim_customers.sql {{ config(materialized='table') }} SELECT c.customer_id, c.customer_name, c.city, COUNT(f.order_id) AS orders, SUM(f.revenue) AS lifetime_value FROM {{ ref('stg_customers') }} c LEFT JOIN {{ ref('fct_sales') }} f ON c.customer_id = f.customer_id GROUP BY 1,2,3

Marts = materialized table. Siap konsumsi untuk dashboard dan reporting.

Praktikum 4

Testing: schema.yml + dbt test

# models/staging/schema.yml version: 2 models: - name: stg_orders columns: - name: order_id tests: - not_null - unique - name: order_status tests: - not_null - accepted_values: values: ['completed', 'pending', 'cancelled', 'unknown'] - name: customer_id tests: - relationships: to: ref('stg_customers') field: customer_id

dbt test — jalankan SEMUA test sekaligus. Gagal = pipeline berhenti, data terjaga.

Konfigurasi

sources.yml — Definisi Data Sumber

# models/staging/sources.yml version: 2 sources: - name: raw description: "Raw data tables dari operational DB" tables: - name: raw_orders description: "Data order mentah — 200K baris" - name: raw_products description: "Katalog produk — 500 items" - name: raw_customers description: "Data pelanggan — 10K records"

sources.yml = kontrak data. Definisikan sumber sekali, gunakan source() di semua staging models.

Konfigurasi

dbt_project.yml — Konfigurasi Project

# dbt_project.yml name: 'online_retail_pipeline' version: '1.0.0' profile: 'online_retail' model-paths: ["models"] test-paths: ["tests"] macro-paths: ["macros"] models: online_retail_pipeline: staging: +materialized: view # staging = view (ringan) intermediate: +materialized: view # intermediate = view marts: +materialized: table # marts = table (cepat query)

Materialization per folder! Staging/intermediate = view (hemat storage), marts = table (cepat query).

Konfigurasi

profiles.yml — Koneksi Database

# ~/.dbt/profiles.yml online_retail: target: dev outputs: dev: type: duckdb path: 'dev.duckdb' # file database lokal threads: 4 # parallel execution prod: type: duckdb path: 'prod.duckdb' # production database threads: 8

DuckDB = zero server. Cukup file .duckdb. Tidak perlu install PostgreSQL, MySQL, atau Snowflake.

Tips &
Kesalahan Umum

04

Hindari

Kesalahan Umum dbt

  • Hardcode table name — selalu pakai ref() dan source(). Fix: ganti FROM table → FROM {{ ref('model') }}
  • Semua di satu model — query 500 baris SQL. Fix: pecah ke staging → intermediate → mart
  • Tidak ada test — data rusak, tidak ada yang tahu. Fix: minimal not_null + unique di primary key
  • Incremental tanpa unique_key — duplikasi data. Fix: selalu set unique_key di config
  • Skip documentation — tim baru bingung. Fix: tulis description di schema.yml
  • Tidak pakai CTE — SQL sulit dibaca. Fix: pakai WITH ... AS untuk setiap langkah

Tips

dbt Best Practices

Naming Convention

stg_ (staging), int_ (intermediate), fct_ (fact), dim_ (dimension). Konsisten!

One Model, One Job

Setiap model punya satu tanggung jawab. Jangan campur cleaning + business logic.

Test Everything

Primary key: not_null + unique. Foreign key: relationships. Status: accepted_values.

Git Workflow

Branch per feature, PR review, CI/CD jalankan dbt test otomatis.

DRY with Macros

Logika berulang? Buat macro Jinja. Reuse di banyak model.

Limit Scope

dbt run --select stg_orders+ → jalankan model tertentu + downstream.
